# Требования: производительность листингов, LCP/TBT, DOM, аналитика, a11y

> Дата: 23 марта 2026  
> Статус: в реализации (часть P0/P1 закрыта в коде; см. чеклист §9)  
> Контекст: отчёты PageSpeed / Lighthouse по категорийным листингам (LCP ~2.9 s, TBT ~400 ms, DOM ~1.6k узлов, Метрика ~260 ms main thread, unused JS, a11y).

Документ задаёт **измеримые NFR**, границы контуров (HTML / API / клиент) и **требования по доступности**, вытекающие из аудитов.

---

## 1. Цели и критерии успеха

### 1.1 Метрики (мобильный лабораторный прогон, условия как PSI: 4G slow)

| ID | Метрика | Цель | Примечание |
|----|---------|------|------------|
| R1.1 | LCP (эталонные URL категорий) | ≤ 2.5 s | Не хуже зафиксированного baseline без регрессии |
| R1.2 | TBT | ≤ 200 s | Учитывать вклад сторонних скриптов отдельно |
| R1.3 | TTFB HTML (p95 на горячих страницах) | ≤ 800 ms или улучшение ≥ 30% к baseline | Согласовать полевые vs лабораторные |
| R1.4 | CLS | ≤ 0.1, не ухудшать | Текущий baseline ≈ 0 — сохранять |
| R1.5 | p95 `GET /api/listing` | Значение X из логов | Задать X после замеров |

### 1.2 Baseline

- Перед изменениями: зафиксировать URL, дату, скрин PSI, при возможности **p95** латентности HTML и `/api/listing`.

## 2. Документ и кэш

- **R3.1** Публичные листинги без персонализации: **edge / ISR-кэш** с явным TTL и инвалидацией при публикации (у `[...slug]` уже `revalidate = 3600` — при необходимости усилить CDN-заголовки на уровне хостинга).
- **R3.2** Не раздувать синхронную работу до первого байта HTML сверх одного батча карточек и необходимых агрегатов.
- **R4** Свести к минимуму **LCP element render delay** (гидратация, блокирующий JS до отрисовки главного контента).

---

## 3. API листинга (`/api/listing`)

- **R5.1** Верхняя граница `limit` задокументирована (в коде cap 60).
- **R5.2** `enrichCardsWithDetails` укладывается в бюджет или для листинга предусмотрен облегчённый путь (отложено до профилирования).
- **R6.1** Rate limiting на публичные API.
- **R6.2** Деградация при ошибке БД без зависания UI.

---

## 4. Клиент: JS, main thread, DOM

- **R7.1** Яндекс.Метрика: не в критической фазе до LCP; загрузка **`lazyOnload`** (или после согласия / idle — по юридическим требованиям).
- **R7.2** Измерять вклад Метрики в main thread; целевое снижение относительно baseline.
- **R8.1** Сократить unused JS на странице листинга: **code splitting** (динамический импорт тяжёлого клиентского каталога).
- **R8.2** При поддержке только современных браузеров — скорректировать `browserslist`, убрать лишние полифиллы (отдельная задача).
- **R9** При росте трафика: виртуализация сетки или ограничение числа узлов в DOM при длинном infinite scroll; устранить заметный **forced layout** в списках.

## 5. Изображения

- **R10** Оптимизация качества/формата через `next/image`, согласованные `sizes`, приоритет только для LCP-кандидатов.

## 6. Доступность (Lighthouse)

- **R11** Иерархия заголовков: **h1** → **h2** → **h3** без пропусков; на листинге категории — секция каталога с **h2** (в т.ч. `sr-only` при необходимости).
- **R12** Контраст текста на белом фоне — не ниже **WCAG AA** (замена `text-zinc-400` для мелкого текста на более тёмные токены).
- **R13** Кнопки-иконки: **`aria-label`** (не только `title`).
- **R14** ARIA: не использовать запрещённые комбинации; для чипа просмотров — валидная роль (например `role="img"` + `aria-label`).
- **R15** Touch targets: интерактивы **≥ 44×44 px** где возможно без поломки дизайна (пилюля переключения группы и т.д.).
- **R16** Не раздувать HTML чрезмерно длинными `aria-label` на каждой карточке (по возможности — краткая метка).

## 10. Страница карточки `/p/[slug]` (аналогичный аудит)

**Было (риски PSI):**

- Два `next/image` с **`priority`** в герое (размытый фон + основное фото) — двойной preload, конкуренция за сеть и LCP.
- Тяжёлый клиентский чанк `CardPageClient` без отдельного code split.
- Мелкие a11y: подписи навигации по фото, «Поделиться», копирование промпта; контраст `text-zinc-400` на белом; лишний `aria-label` на `<p>` со счётчиком просмотров (текст и так виден).

**Сделано в коде:**

- Фон blur: раньше второй `<img>` забирал LCP; теперь **только после загрузки героя** и через **CSS `background-image`** (без `<img>`).
- Герой (основное фото): **`priority`**, **`fetchPriority="high"`**, **`decoding="async"`**.
- **`dynamic()`** для `CardPageClient` в `p/[slug]/page.tsx` + компактный skeleton.
- Breadcrumb **`text-zinc-500`**; подписи промптов и футер-блок — контрастнее; кнопки с **`aria-label`**; русские подписи стрелок.

**Бэклог (при необходимости):**

- Отложить `POST /api/card-view` до `requestIdleCallback` (микрооптимизация TBT).
- Перенести JSON-LD в `next/script` с `afterInteractive` (единообразие; на LCP почти не влияет).
