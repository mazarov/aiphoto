# Steal This Vibe: несколько пользовательских фото (UI + UX)

> **Статус:** реализовано в extension (`userPhotos`, до 4 снимков)  
> **Дата:** 2026-03-23  
> **Область:** Chrome extension — side panel (`extension/sidepanel/*`)

## 1. Зачем

Один снимок пользователя не всегда даёт модели достаточно сигнала об идентичности (ракурс, освещение). Бэкенд уже принимает **массив** путей в Storage и в `generate-process` последовательно кодирует **все** загруженные фото пользователя в parts для Gemini (см. `landing/src/app/api/generate/route.ts`: **от 1 до 4** фото). Панель сейчас передаёт только **`photoStoragePaths: [singlePath]`** — нужно выровнять UI с возможностями API и дать пользователю осознанно добавить **2–4** снимка **одного** человека.

## 2. Ограничения API (источник правды)

| Правило | Значение |
|--------|----------|
| Минимум | 1 фото |
| Максимум | **4** фото (`POST /api/generate` → `validation_error` при `> 4`) |
| Форматы | как у текущего upload: jpeg / png / webp |
| Порядок | порядок в массиве = порядок image parts после метки субъекта (B) в мультимодальном запросе |

## 3. Техническая модель (для реализации)

- **Состояние:** заменить или дополнить `photoStoragePath` + `uploadedFileName` на список слотов, например  
  `userPhotos: Array<{ storagePath: string; fileName: string; previewObjectUrl?: string }>`  
  (preview URL только в памяти; после перезагрузки — signed URL по каждому `storagePath`, аналогично текущему одному фото).
- **Upload:** каждый файл — отдельный `POST /api/upload-generation-photo` (как сейчас); в список добавляется новый элемент до лимита 4.
- **Generate:** `photoStoragePaths: state.userPhotos.map(p => p.storagePath)` (или эквивалент).
- **Валидация перед запуском:** `length >= 1`; кнопка генерации disabled при 0 фото.
- **Persist:** сериализовать массив в `localStorage` (ключ состояния панели), миграция со старого формата: если есть только `photoStoragePath` — превратить в массив из одного элемента.
- **Сброс сессии:** очистить весь массив превью (revoke object URLs).

## 4. Критерии приёмки

1. Пользователь может добавить **второе, третье, четвёртое** фото без удаления первого (до 4).
2. Видит **мини-превью** каждого загруженного фото и имя файла (или короткий label).
3. Может **удалить** слот (крестик) до запуска пайплайна; при 0 фото — явное состояние «добавьте фото».
4. Опционально (nice-to-have в той же итерации): **смена порядка** drag-and-drop или кнопки «вверх/вниз» — порядок должен совпадать с порядком в `photoStoragePaths`.
5. Один запуск STV отправляет в `/api/generate` **все** пути одним массивом.
6. Копирайт рядом с зоной загрузки объясняет: *один человек, разные ракурсы / освещение, до 4 снимков*.

## 5. Вне скоупа (v1)

- Отдельный лимит кредитов за количество фото (сейчас списание за модель, не за число B-images — не менять без продуктового решения).
- Загрузка **>4** файлов за раз через `multiple` на input без отсечения — лучше явно показать «добавлено 4 из 4» и не принимать лишнее.

## 6. Продуктовый и визуальный дизайн (роль Startup Product Designer)

Ниже — прогонка структуры из `@startup-designer`: как должно **ощущаться** UI в боковой панели, без привязки к конкретным пикселям в коде.

### 6.1 Product understanding

**Продукт:** расширение «Steal This Vibe» — пользователь копирует визуальный стиль с референс-сайта и переносит его на **себя**, загружая своё фото (скоро — несколько).  
**Проблема:** один кадр часто недостаточен для узнаваемости лица/тела; пользователь не понимает, что можно (и полезно) дать модели больше контекста.  
**Цель UX:** сделать мульти-фото **очевидным, необязательным усилением**, а не усложнением обязательного шага — чтобы первый запуск по-прежнему «одно фото → готово», а продвинутый сценарий чувствовался премиально и под контролем.

### 6.2 Structure (секции панели в зоне «Ваше фото»)

1. **Заголовок секции** — без смены смысла: «Ваше фото» / локализация как сейчас.  
2. **Подзаголовок одной строкой** — зачем несколько снимков (см. микрокопирайт).  
3. **Сетка превью** (1–4 ячейки) — единый визуальный ритм.  
4. **Первичное действие** — «Добавить фото» / при пустой панели — крупная зона выбора файла.  
5. **Вторичное** — при достижении лимита: нейтральный текст «Максимум 4 фото».  
6. **Связь с пайплайном** — индикатор готовности: «Готово к генерации» когда `≥1` фото загружено.

### 6.3 Текстовый макет блока (как выглядит в UI)

Ниже — «нарисовано» символами: ширина условно как у узкой side panel (~320–360px). Квадратные ячейки — превью фиксированного размера (например 72–88px сторона), скругление как у остальных карточек панели.

---

**А) Пустое состояние — ни одного фото**

```
┌─────────────────────────────────────┐
│  Ваше фото                          │  ← заголовок секции (semibold)
│  Один человек — до 4 снимков         │  ← подзаголовок, muted, 12–13px
│  (разные ракурсы помогают модели)    │
│                                     │
│  ┌─────────────────────────────┐   │
│  │                             │   │
│  │            +                │   │  ← пунктирная рамка 1px, фон #f5f5f7
│  │      Выберите фото          │   │
│  │                             │   │
│  └─────────────────────────────┘   │
│                                     │
│  Например: крупный портрет + рост   │  ← ещё одна строка muted
└─────────────────────────────────────┘
```

Вся пунктирная зона кликабельна (как сейчас label на `input type=file`). Курсор — pointer.

**Б) Одно фото загружено**

```
┌─────────────────────────────────────┐
│  Ваше фото                          │
│  Один человек — до 4 снимков         │
│                                     │
│   ┌──────────┐                      │
│   │ [превью] │  IMG_2401.jpg        │  ← имя файла обрезать с … если длинное
│   │   фото   │  [×]                 │  ← крестик в углу превью (не на тексте)
│   └──────────┘                      │
│                                     │
│  [ + Добавить ещё ]                  │  ← вторичная кнопка, outline / ghost
│                                     │
│  Загружено 1 из 4                   │  ← опционально, очень мелко muted
└─────────────────────────────────────┘
```

Кнопка «Добавить ещё» открывает тот же file picker (второй скрытый `input` или один и тот же — на усмотрение разработки).

**В) Два фото — сетка 2 колонки**


Между ячейками зазор 8px. Обе строки имён можно сделать одной высоты с `text-overflow: ellipsis`.

Кнопка «Добавить ещё» **скрыта** или **disabled** с тем же текстом «Максимум 4 фото» рядом — на выбор; главное — не давать пятый upload.

**Д) Во время загрузки очередного файла**

Первая карточка остаётся как есть; вторая не кликабельна для удаления пока upload не завершён (или можно отменить — по продукту).

Визуально: те же `border-radius` у превью пользователя, что у превью референса выше (если оно есть), чтобы пара «стиль / вы» читалась парой.

### 6.4 Detailed UI layout (детали реализации)

- **Пустое состояние:** одна карточка-dashed с иконкой «+» и текстом «Выберите фото»; ниже мелким — *«Можно до 4 снимков одного человека»*.  
- **1 фото:** превью квадрат/слегка rounded; под сеткой кнопка **outline** «Добавить ещё» (не конкурирует с основным CTA запуска ниже по панели).  
- **2–4 фото:** сетка **2×2**, без горизонтального скролла.  
- **Карточка превью:** миниатюра, обрезка `object-fit: cover`; в углу **иконка закрытия** (touch target ≥ 24px); при hover — лёгкая подсветка рамки.  
- **Порядок (если делаем reorder):** иконка «⋮⋮» или стрелки; после перестановки порядок превью = порядок в запросе.  
- **Загрузка:** на карточке skeleton или тонкий progress; блокировка повторного выбора того же файла опционально.

### 6.5 Visual style

- **Типографика:** подзаголовок секции на ступень меньше основного заголовка панели; микрокопирайт — `muted`, 12–13px, достаточный line-height.  
- **Цвет:** превью на нейтральном фоне `#f5f5f7` / граница `1px` subtle; активная рамка при фокусе — акцентный цвет бренда панели (как у существующих кнопок).  
- **Отступы:** единый `gap` между превью (8px); секция не «раздувается» — максимальная высота блока превью ограничена (например ~220px на всю сетку 2×2).  
- **Премиальность:** никаких ярких бейджей «NEW»; спокойная подача, как у Linear/Notion empty states.

### 6.6 UX details

- После успешной загрузки — **краткий toast** «Фото добавлено» (как сейчас), с учётом множественного числа в i18n.  
- При попытке добавить 5-е фото — **мягкий** toast «Не больше 4 фото», без модалки.  
- Удаление последнего фото — возврат к пустому состоянию; если пайплайн уже был в progress — не применимо (сессия обычно сбрасывается явно).  
- **Accessibility:** `label` для file input; кнопка удаления с `aria-label` («Удалить фото»).

### 6.7 Design improvements (чтобы ощущалось дороже)

- **Один человек — один визуальный «стек»:** лёгкая подпись *«Все снимки — один человек»* снижает риск заливки разных людей.  
- **Подсказка-пример:** одна строка *«Например: лицо крупно + в полный рост»* — конвертирует абстракцию в действие.  
- **Согласованность с референсом:** если выше показывается превью source-сайта, сетка пользовательских фото визуально **параллельна** (те же скругления), чтобы читалось как «A = стиль, B = вы (несколько кадров)».
