# Требования: канонические пресеты размеров фото (лендинг)

> Дата: 2026-03-23  
> Статус: реализовано — `landing/src/lib/card-image-presets.ts`, `getStorageCardMediaUrl` в `landing/src/lib/supabase.ts`  
> Контекст: единые размеры отдачи и подсказок `next/image`, без «зоопарка» значений в компонентах; опционально связка с Supabase `render/image` / imgproxy на self-hosted.

---

## 1. Цель

1. **Один источник правды** по числовым размерам для **промо-фото карточек** (Storage / `render/image` / параметры оптимизатора).
2. В UI — не более **двух именованных пресетов** для этих фото; все текущие и будущие места отображения маппятся на пресет **A** или **B**.
3. Исключения (см. §5) перечислены явно и не размножаются без обновления этого документа.

---

## 2. Термины

| Термин | Смысл |
|--------|--------|
| **Физическая ширина** | Пиксели по длинной стороне (или `width` в URL трансформации), согласованная с `devicePixelRatio` (минимум: покрыть **2×** для типичного CSS-размера пресета). |
| **`sizes` (Next.js)** | Подсказка браузеру для выбора варианта из `srcset`; должна быть согласована с layout, но **строки `sizes` не должны плодиться** — только константы из общего модуля. |
| **Пресет** | Именованная пара: `{ maxPhysicalWidth, sizesExpression }` (и при необходимости `quality`, `resize` для imgproxy). |

---

## 3. Пресеты (зафиксировать числа при реализации)

До имплементации команда выбирает конкретные числа под макет и PSI; в коде они живут **только** в одном файле, например `landing/src/lib/card-image-presets.ts`.

### 3.1 Пресет **A — сетка и мелкие врезки** (`CARD_GRID`)

Использовать **везде**, где картинка карточки занимает **часть экрана** или **миниатюру**:

- листинг: `PromptCard`, `GroupedCard` (основное фото и фоновая карточка стека);
- категории: `CategoryCard`;
- поиск: превью в `SearchBar`;
- карусель: **миниатюры** в `PhotoCarousel`;
- оверлеи «before» / мелкие врезки на карточке: `PromptCard`, `GroupedCard`, `CardPageClient` (before-strip);
- круглые чипы вариантов на `/p/[slug]` (`CardPageClient`, 16×16 CSS) — **тот же пресет A по URL**, масштабирование CSS (простота важнее экономии байт на редких 16px).

**Требование:** одна **физическая максимальная ширина** (например 640 или 768 px — выбрать и зафиксировать) для всех перечисленных мест.

### 3.2 Пресет **B — герой детальной страницы** (`CARD_HERO`)

Использовать только там, где промо-фото **главный контент** и задаёт LCP:

- основное изображение на `/p/[slug]` (`CardPageClient` / карусель **главный** слайд, не strip);
- при необходимости — полноширинный режим `PhotoCarousel` на десктопе, если он визуально существенно крупнее сетки.

**Требование:** вторая **физическая максимальная ширина** (например 960–1280 px — выбрать одну).

### 3.3 Соответствие `sizes`

- Для пресета A — **одна** общая строка `SIZES_CARD_GRID` (или генератор от breakpoints), используемая во всех компонентах пресета A.
- Для пресета B — **одна** строка `SIZES_CARD_HERO`.
- Запрещено вносить в компоненты локальные варианты вроде отдельных `260px` / `300px` без изменения §3 и модуля пресетов.

**Текущий зоопарк (к удалению при имплементации):** разные `sizes` в `PromptCard`, `GroupedCard`, `CategoryCard`, `PhotoCarousel`, `CardPageClient`, `SearchBar` — см. grep `sizes=` в `landing/src/components/`.

## 4. Поставка URL (Storage / imgproxy / Next)

1. **Если включён `render/image`:** функция сборки публичного URL для карточек принимает **`preset: 'A' | 'B'`** и подставляет **только** разрешённые ширину/качество из того же модуля, что и `next/image`.
2. **Если остаётся только `next/image` (Vercel):** физический размер всё равно задаётся из того же модуля (через `width`/`sizes` и при custom loader — те же числа).
3. **Запрет:** хардкод `width=500` в одном месте и `width=600` в другом для одного и того же пресета A.

---

## 5. Исключения (не входят в «1–2 пресета карточки»)

| Источник | Почему вне пресетов A/B |
|----------|-------------------------|
| Аватар пользователя (`HeaderClient`, OAuth URL) | Не наш Storage; фиксированный маленький `width`/`height` в одном месте. |
| Логотип (`SiteLogoMark`) | SVG/статика, не карточный pipeline. |
| Результаты web-генерации / произвольные URL | Отдельная политика при появлении массового показа в UI. |

Новые исключения — только с обновлением этого раздела и архитектурной заметки в `docs/architecture/01-landing.md`.

## 6. Нефункциональные требования

| ID | Требование |
|----|------------|
| N1 | Любой новый компонент с фото карточки импортирует пресеты из `landing/src/lib/card-image-presets.ts` (или согласованное имя), без литералов ширины. |
| N2 | В ревью: изменение числового размера — **один diff в модуле пресетов** + прогон PSI на 1 листинге и 1 `/p/[slug]`. |
| N3 | LCP: пресет B не снижеет приоритет героя; пресет A не раздувает `srcset` сверх необходимого (не добавлять третий физический размер без решения из §7). |
| N4 | Документация: после внедрения — обновить `docs/architecture/01-landing.md` (параграф про изображения и пресеты). |

---

## 7. Опциональное расширение (не по умолчанию)

Если метрики покажут, что **16px чипы** или **48px поиск** тянут слишком много байт с пресета A:

- ввести **микропресет A-** (третий физический размер) **только** для этих двух случаев;
- зафиксировать в §3 отдельным подпунктом и с лимитом использования (не более N мест в коде).

До появления данных — остаёмся на строго **A + B**.

---

## 8. Критерии приёмки

1. Grep по `landing/src`: нет строк `sizes="` с уникальными литералами вне `card-image-presets.ts` (допускаются исключения из §5).
2. Любой URL `render/image` для `prompt_card` строится через общий хелпер с `preset: 'A' | 'B'`.
3. Зафиксированы выбранные числа для A и B в комментарии в модуле пресетов + ссылка на этот документ.
4. Нет регрессии LCP на эталонных URL из `docs/23-03-listing-performance-requirements.md` (сравнение до/после).
