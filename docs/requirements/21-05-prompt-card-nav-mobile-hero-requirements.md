# Требования: навигация по листингу на `/p/[slug]` и mobile-first «герой фото»

> **Дата:** 2026-05-21  
> **Статус:** черновик требований (реализации нет — этот файл задаёт объём работ).  
> **Маршрут:** страница карточки промта **`/p/[slug]`** (`landing/src/app/p/[slug]/page.tsx`, клиент **`CardPageClient`**).  
> **Референс макета (mobile):** скриншот от 2026-05-21 (ориентир: полноэкранное hero-фото, верхняя полоска сегментов, нижний градиент, вторичная полоска маленьких действий и один крупный primary CTA). Использовать **только** как подсказку по композиции и иерархии; текст, иконки и бренд стороннего приложения переносить не нужно.

## 1. Продукт и задача пользователя

**Проблема сегодня:** на странице карточки пользователь теряется в потоке «вернуться в листинг и выбрать соседа» и на мобильном недостаточно визуального акцента на результате промта (фото).

**Цели после изменений:**

| Цель | Как понимаем успех |
|------|---------------------|
| **G1 — соседние карточки** | С листинга пришёл в `/p/[slug]` — может **перейти к предыдущей/следующей карточке из того же упорядоченного списка**, **в той же вкладке**, не возвращаясь в сетку и **не открывая каждый переход новой вкладкой**. |
| **G2 — мобильный hero** | На узких вьюпортах первый экран **занимает главное фото** (полноэкранно или почти полноэкранно), текст и действия **наложены/смещены вниз**, читаемость за счёт градиента/полупрозрачных блоков как в референсе; **поведение существующих кнопок не ломаем**. |

**Жёсткое ограничение (прод-позиционирование):**

- **Prev/next — только текущая вкладка:** клики по стрелкам **не** должны каждый раз открывать новую карточку в отдельной вкладке/окне; см. §2.2.
- **Не менять функционал страницы:** те же промты, те же блоки генерации/STV если есть, те же действия LexyGPT / «Копировать» / реакции / избранное и т.д. — допускается **только смена layout, порядка блоков на мобильном и добавление prev/next**.
- Изменений **бекенда и контрактов API** этим документом **не предполагаем** (источник истории списка — клиент после реализации, см. §2.1–2.2).

## 2. Навигация «назад / вперёд» по контексту листинга

### 2.1 Область навигации (scope списка)

**Определение «того же листинга»** — нужно явно восстанавливать **упорядоченный набор slug’ов карточек**, из которых пользователь перешёл на `/p/...`.

Минимально корректные варианты (выбрать один на этапе реализации, второй считать fallback):

| Вариант | Описание | Плюсы / минусы |
|---------|----------|----------------|
| **A — `sessionStorage` + referrer** | При рендере листинга (страница каталога / тег и т.д.) сохранять **массив slug** текущего запроса (порядок как в RPC/сортировке видимых ячеек) + **индекс/позицию текущего открытия** при переходе на `/p/[slug]` | Простота, без URL-шума; очищается при закрытии вкладки. |
| **B — Query в URL карточки** | При клике с листинга: `/p/slug?from=listing&cursor=abc` или зашифрованный указатель набора | Делится ссылкой без «контекста»; сложнее. |

**Требование:** реализовать **хотя бы A** так, чтобы навигация работала после типичного пути пользователя: **из сетки → карточка → prev/next**.

**Не в scope навигации (v1):**

- Переходы из внешних ссылок, поисковой выдачи, «скопировать URL» без визита листинга в этой сессии → **стрелки не показывать или disabled** с понятной подписью («Нет последовательности из листинга» tooltip / visually hidden текст).
- Склейка с **Infinite scroll** второй страницы: желательно, но можно **iteration 2** — в v1 допустим список только «видимые + подгруженные» slug’ы, зафиксированные при уходе с листинга.

### 2.2 Одна вкладка браузера (обязательно)

Каждый переход предыдущая / следующая карточка по стрелкам (**и по жесту, если добавят**) — только **«мягкая» навигация в приложении**, **тот же `window`/вкладка**; пользователь продолжает работу **на той же странице-карточке** как на одном последовательном просмотрщике, без «горы вкладок».

- Использовать **`router.push` / клиентская смена маршрута** (App Router); **не** открывать соседа в **`target="_blank"`**, **`window.open('/p/...')`**, **среднем клике по ссылке** на переход между соседями. Элементы prev/next предпочтительно как **`<button type="button">`**, чтобы не давать браузеру стандартное «Открыть в новой вкладке» для ссылки.
- Ссылка **`/p/[slug]` из листинга** может по-прежнему иметь **`target="_blank"`** там, где это уже задано **продуктом** для сетки — это **отдельный UX** от стрелок на уже открытой карточке.

### 2.3 Поведение по клику / touch

| Платформа | UI |
|-----------|-----|
| **Desktop** | Видимые **стрелки влево/вправо** в зоне **hero-изображения** (или компактная полоса у краёв), клики не должны блокироваться нижним fixed-баром. |
| **Mobile** | Те же **стрелки** **или** **swipe-жесты** между карточками (**опционально v1.5** после базовых стрелок). Если жест включён — обязательны **hints**/`aria` и конфликт с горизонтальными стрелками «фото внутри карточки» не должен открыть соседа по ошибке. |

**Фокус и a11y:**

- Стрелки — кнопки с **`aria-label`** («Предыдущая карточка из листинга» / «Следующая…»).
- При смене slug — **обновление заголовка**, **перенос фокуса** на ключевой landmark; навигация — **одна вкладка**, по умолчанию **клиентский переход** (`router.push`); полная перезагрузка допустима только как осознанный технический fallback, не как норма.

### 2.4 Состояние и краевые случаи

- **Первый элемент списка** — назад недоступен; **последний** — вперёд недоступен (disabled или скрыть соответствующую сторону по продукту).
- Если список **известен**, но текущего **slug нет в списке** (рассинхрон данных) → prev/next скрыть, **не падать**.
- При **переходе на соседнюю карточку** — **те же сохранённые query/scroll policy**, что считаем нужным для чистого UX по умолчанию: **прокрутка в начало страницы** для новой карточки.

## 3. Изменение mobile UI под «герой как в референсе»

### 3.1 Визуальные акценты (без смены логики)

На брейкпоинте **до `md` (или совпадающем с текущими mobile-правилами лендинга)**:

1. **Фото главной карточки** — доминирующий элемент: **full-bleed по ширине**, по высоте **значительная доля viewport** (ориентир: **≥60–75% высоты** первого экрана при одном фото; при нескольких фото сохранить **внутрикарточную** галерею, но общий паттерн — «сторис-подобный» верх + контент поверх нижнего градиента).
2. **Нижняя область первого экрана** — **вертикальный градиент** от прозрачного к ~40–65% затемнённого (как референс), чтобы текст/кнопки читались на светлых участках изображения.
3. Вторичная полоса: **маленькие pill-кнопки** в один ряд (иконки + короткий текст) для **действий, которые уже есть** на странице (например «Поделиться», если уже реализовано; **не** добавлять «Скачать» только ради параллели с Lexy-сервисом, если этого нет в текущей странице).
4. Основная зона промта / LexyGPT / Копировать — располагается **ниже первого экрана** или **в нижнем оверлее** с тем же набором кнопок, что уже есть (порядок и grouping — см. следующий абзац).

### 3.2 Согласование с текущими компонентами

- **`CardPageClient`:** перестановка блоков допускается **только** для mobile; desktop layout может сохраниться ближе к текущему (или слегка унифицироваться без удаления возможностей).
- **Sticky-бар LexyGPT + «Копировать»** не должен конфликтовать с **стрелками prev/next** (z-index, safe-area): зафиксировать в QA.
- Индикация **нескольких фото текущей карточки** (точки/черточки в референсе): если продукт согласен — **тонкая шкала-сегмент** сверху **только** для галереи **внутри** карточки; **не** путать с прогрессом prev/next между карточками.

### 3.3 Нефункциональные требования

- **LCP:** сохранить приоритет отдачи главного изображения (см. `docs/architecture/01-landing.md`, §`/p/[slug]`).
- **`prefers-reduced-motion`:** минимизация параллакса/тяжёлой анимации для hero-слоя.
- **Contrast:** текст на градиенте не ниже WCAG AA для вторичных подписей; при необходимости — усиление градиента или полупрозрачная подложка под текст.

## 4. Критерии приёмки (Checklist)

- [ ] С листинга открыта карточка — **видны работающие prev/next**, переход загружает другую карточку того же упорядоченного набора **в этой же вкладке** (число открытых вкладок не растёт).
- [ ] Открытие **`/p/...`** из адресной строки без контекста — **нет активной навигации** или понятное disabled-состояние.
- [ ] **Mobile-first hero** соответствует §3 по ширине/высоте и градиенту (визуальное ревью с референсом).
- [ ] Любая существующая кнопка/флоу (**Lexy, копирование, реакции, избранное, автор и т.д.**) выполняется **как до изменений**.
- [ ] Стрелки prev/next не перекрыты fixed UI и имеют понятный **feedback** hover/active/focus.
- [ ] Нет регресса Lighthouse LCP страницы карточки сверх допустимого порога (зафиксировать численно перед стартом, например PSI mobile).

## 5. Открытые вопросы (зафиксировать до разработки)

1. Точное **название ключей `sessionStorage`** и максимальный **размер сохранённого списка** (обрезка при длинном infinite-scroll).
2. Нужно ли показывать **индекс** «12 из 487» между стрелками.
3. **Групповые карточки split:** prev/next между **вариантами одной группы** vs **обычными соседями сетки** — одна ли это навигация или две области ответственности.
4. Синхронизация с **`router.refresh()`** после действий пользователя на карточке (видимость черновика) — не должна терять массив из `sessionStorage`.
