# ТЗ: непрозрачные пути в Storage + миграция (вариант B)

> Выбранный подход: **вариант B — миграция Storage и БД** для уже существующих объектов; новые загрузки сразу по новому шаблону.

## 1. Цель

**Проблема:** публичные URL изображений содержат сегмент `telegram/{source_dataset_slug}/…`, где slug совпадает с именем папки Telegram HTML-экспорта (например `bananogenpromt_ChatExport_2026-03-13`).

**Цель:**

1. Для всех релевантных медиа в bucket `prompt-images` хранить путь без человекочитаемого slug датасета.
2. **Перенести** существующие объекты на новые ключи и **обновить** записи в БД так, чтобы лендинг и внешние потребители получали только новые URL (после миграции и очистки старых ключей — старые прямые ссылки на файлы перестанут работать; см. §8).

**Вне scope:**

- Смена URL страниц карточек (`/p/{slug}`).
- Удаление или смысловая замена поля `prompt_cards.source_dataset_slug` (остаётся для внутренней логики).
- Обеспечение 301 с уровня Supabase Storage «из коробки» (при необходимости — отдельная инфраструктура).

## 2. Текущее состояние

### 2.1. Шаблон пути (legacy)

- Bucket: `prompt-images`.
- Шаблон: `telegram/{dataset_slug}/{source_message_id}/{card_split_index}/{media_index}{ext}`.

**Код:** `src/ingest-telegram-export-to-supabase.ts` (`uploadMedia`), `src/backfill-missing-photos.ts`, `src/reupload-missing-media.ts`.

### 2.2. Потребление

- `prompt_card_media.storage_bucket` + `prompt_card_media.storage_path`.
- При наличии записей — `prompt_card_before_media` с тем же паттерном `(bucket, path)` и ограничением `UNIQUE(storage_bucket, storage_path)`.

### 2.3. Идентификатор датасета

- Таблица `import_datasets`: `id` (uuid), `dataset_slug` (уникальный).
- Связь карточки с датасетом: `prompt_cards.source_dataset_slug` соответствует `import_datasets.dataset_slug` (для резолва `id` при миграции).

---

## 3. Целевой шаблон пути

**Канонический шаблон:**

```text
telegram/ds/{import_dataset_id}/{source_message_id}/{card_split_index}/{media_index}{ext}
```

- `{import_dataset_id}` — UUID из `import_datasets.id` (формат с дефисами, как в Postgres/Supabase).
- Остальные сегменты без изменений относительно legacy-шаблона.

**Инварианты:**

- Локальное чтение файлов при инжесте с диска по-прежнему по `card.datasetSlug` + относительный путь медиа; **ключ в Storage не строить из slug папки экспорта**.
- Один `import_datasets.id` → один префикс `telegram/ds/{id}/` для всех объектов этого датасета.

## 5. Функциональные требования — миграция B (Storage + БД)

### FR-B1. Область данных

- Обработать все строки, где:
  - `storage_bucket = 'prompt-images'`, и
  - `storage_path` соответствует legacy-паттерну: первый сегмент после `telegram/` **не** равен литералу `ds` **или** второй сегмент не является валидным UUID в каноническом виде (уточнить в реализации одним правилом: «не начинается с `telegram/ds/{uuid}/`»).

Таблицы минимум:

- `prompt_card_media`
- `prompt_card_before_media` (если есть пути в том же bucket с legacy-префиксом)

### FR-B2. Вычисление нового пути

Для каждой строки медиа:

1. По `card_id` получить `prompt_cards.source_dataset_slug`.
2. По `dataset_slug` получить `import_datasets.id` (join или отдельный lookup; при отсутствии — запись в отчёт ошибок и **skip** с ненулевым exit code по итогу прогона или флагом `--strict`).
3. Новый путь: заменить префикс  
   `telegram/{любой_legacy_segment}/`  
   на  
   `telegram/ds/{import_dataset_id}/`,  
   сохранив хвост `{source_message_id}/{card_split_index}/{media_index}{ext}` **если** он совпадает с текущим legacy-путём после первого сегмента; иначе парсить путь по сегментам и собирать заново (зафиксировать в коде единый парсер сегментов).

**Важно:** если фактическая структура пути отличается от ожидаемой (доп. сегменты), мигратор должен логировать и не ломать данные вслепую.

### FR-B3. Операции в Storage

- Для каждой пары `(old_path, new_path)`:
  - если объект по `new_path` уже существует — политика: **skip upload**, только при необходимости выровнять БД (идемпотентность);
  - иначе **скопировать** объект `old_path` → `new_path` (предпочтительно copy для возможности отката до удаления старых ключей);
  - после успешного копирования и обновления БД (см. ниже) — по флагу `--delete-legacy` удалить `old_path`.

Порядок по умолчанию: **сначала copy + update DB, удаление legacy — отдельным проходом или флагом** после выборочной верификации.

### FR-B4. Обновление БД

- Обновлять `storage_path` (и при необходимости связанные кэши не трогать — RPC читают из таблиц).
- Учитывать `UNIQUE(storage_bucket, storage_path)`: перед update проверять, что новый path не занят другой строкой; при конфликте — лог + стратегия (ручной разбор / abort).

Рекомендуемый порядок в одной логической единице работы:

1. Copy в Storage на `new_path` (если ещё нет).
2. `UPDATE ... SET storage_path = new_path WHERE id = ... AND storage_path = old_path` (оптимистичная защита от гонок).

Транзакция с Supabase Storage в одной SQL-транзакции недоступна — фиксировать в runbook: при падении после copy но до update — повторный прогон идемпотентен; при update без copy — ошибка и ретрай.

### FR-B5. CLI / режимы

Обязательные флаги (имена можно уточнить в реализации):

- `--dry-run` — только отчёт `old → new`, без Storage и без UPDATE.
- `--limit` / `--offset` или курсор по `id` — батчи.
- `--delete-legacy` — удалять старые ключи после успешного update (опционально по умолчанию выкл).

### FR-B6. Отчёт

- По завершении: количество обработанных, пропущенных, ошибок; список проблемных `id` / путей / причин.

### FR-B7. Верификация после миграции

- Выборочно: HEAD/GET нового public URL; сравнение размера или checksum при наличии API.
- SQL: нет строк с `prompt-images` и path не matching новый префикс (для согласованного regex).

## 6. Нефункциональные требования

- Резолв `slug → import_datasets.id`: кэш в памяти на процесс, без N+1 на каждую строку медиа.
- Секреты только из env; не логировать service role.
- Батчи и rate limits: не положить Storage/API (настраиваемая задержка между батчами).

---

## 7. Критерии приёмки

1. Код инжеста и вспомогательных скриптов пишет новые объекты только по шаблону §3.
2. Скрипт миграции B с `--dry-run` выдаёт полный план без побочных эффектов.
3. Прогон миграции без `--dry-run`: все целевые строки обновлены; объекты доступны по новым ключам.
4. После прогона с `--delete-legacy` legacy-ключи отсутствуют (или явно задокументировано иное).
5. `UNIQUE(storage_bucket, storage_path)` не нарушен.
6. Обновлён `docs/architecture/03-pipeline.md`.

---

## 8. Риски (вариант B)

- **Прямые ссылки** на старые URL (поиск картинок, соцсети, CSV) дадут **404** после удаления legacy — заложить окно dual-read (не удалять старые объекты N дней) или отдельный редирект-слой, если критично.
- **Длительность** прогона на большом объёме — планировать окно и мониторинг ошибок Storage.
